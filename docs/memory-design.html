🌸 设计概述

记忆系统采用 多智能体架构，将记忆管理从对话 Agent 中完全解耦。

对话 Agent（澪号）专注角色扮演与对话，不感知记忆的存储结构和文件组织。

记忆 Agent 是一个独立 Pi session，只在需要时启动，完成任务后立即清除上下文关机。

📚 三层记忆架构
层级触发格式生命周期访问范围
规则记忆 每次对话拼入 prompt rules/rules.md 永久（手动编辑） 所有会话
短期记忆 上下文超阈值 → compaction short-term/<id>-N.md 会话内，归档后删除 当前会话（代码约束）
长期记忆 用户 UI 归档会话 long-term/<id>.md + index.json 永久 所有会话

🤖 记忆 Agent 详解

✧ 启动与销毁

记忆 Agent 通过 ctx.newSession() 创建独立 Pi session。

每次压缩/归档任务一个 session，agent_end 后立即结束，不留上下文。

♡ 用完即毁，下次有任务再新起。完全无状态。

✧ 工具与权限

工具权限说明
read只读读取对话片段、旧 .md 文件、index.json
write_memory受控写入传入内容 → Python 校验 → 通过才落盘。无 write 权限

♡ Agent 摸不到文件系统。Python 是唯一的写入入口。

✧ 输入与输出

场景输入输出
短期压缩 当前会话原始对话片段 short-term/<id>-N.md
长期归档 全部 short-term/<id>-*.md long-term/<id>.md + index.json 追加条目

✧ 校验内容

类别检查项
Schema每个 index entry 必含 id / date / summary / file / tags / importance
格式date 为 ISO 格式；summary ≤ 50 字；importance 为 1-5 整数
一致性file 指向的 .md 存在且非空；id 不重复；旧条目未被修改
文件.md 文件 >0 且 <500KB；内容可读

校验失败 → 返回具体错误原因 → 记忆 Agent 修正后重调。最多 3 次。

✧ 错误分类

类型处理是否消耗 3 次配额
格式错误返回错误原因 → LLM 重写 → 重试✓ 消耗
基础设施API 超时/429、磁盘满、权限… 单独重试✗ 不消耗
上下文问题对话过长溢出、内容不可理解✓ 消耗

✧ 失败处理

3 次格式校验均失败 → 放弃。

原始对话 / short-term .md 保留不删。

通知前台 "⚠ 记忆整理失败，下次归档时重试"。

不使用降级策略 —— 不留脏数据。

🔍 记忆检索

✧ 检索方式：大模型自主决策

pa-mio 不写关键词匹配逻辑。澪号的 systemPrompt 中包含以下指令：

你有长期记忆，存放在 ~/.personal-agent/memory/long-term/ 检索方式： 1. 先读 index.json 了解有哪些记忆条目及其摘要 2. 根据当前对话内容判断需要读哪些 .md 3. 不需要全部读完，只读相关的那几个 你有短期记忆，存放在 short-term/<当前会话id>-*.md - 可能有多个文件（-1、-2、-3…），按序号读 - 只能读自己当前会话的短期记忆 - 需要其他会话的信息 → 查长期记忆

♡ 代码层拦截 short-term/ 跨会话读取 —— tool_call 事件中检查路径不含当前 session ID 即 block。

📌 规则记忆

规则记忆是 rules/rules.md，手动编辑，每次对话无条件拼入 prompt。

用于存放：

♡特定行为规则（如"晚上 10 点后说话小声一点"）
♡定时事项（如"每周五提醒我清缓存"）
♡永久性的用户偏好

拼入位置：Slot 0 之后、身份层之前，优先级高于身份默认行为。

♡ 设计原则
1.解耦 —— 记忆 Agent 独立于对话 Agent，不共享上下文
2.不解耦不写 —— Agent 只调用 write_memory，Python 校验后落盘
3.只追加不修改 —— index 只增条目，旧 .md 不动
4.失败不降级 —— 格式错误 3 次放弃，原始数据保留，下次重试
5.不阻塞对话 —— 记忆任务后台执行，前台进度可见
6.代码约束优先 —— 跨会话短期记忆读取由代码拦截，不依赖 prompt 约束
