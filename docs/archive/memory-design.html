记忆 Agent 详解

✧ 启动与销毁

通过 ctx.newSession() 创建独立 Pi session。每次压缩/归档一个任务对应一个 session。

agent_end 后立即结束，不留上下文。下次有任务再新起——完全无状态。

✧ 工具与权限

工具权限说明
read只读读取对话片段、旧的 .md 文件、index.json
write_memory受控写入传入内容 → Python 校验 → 通过才落盘。Agent 没有 write 权限

♡ Agent 摸不到文件系统。Python 是唯一的写入入口，不通过校验的内容永远不会落盘。

✧ 输入 & 输出

场景输入输出
短期压缩当前会话原始对话片段（有长度上限）short-term/<id>-N.md
长期归档该会话全部 short-term/<id>-*.mdlong-term/<id>.md + 在 index.json 末尾追加一条

✧ 校验内容

类别检查项
Schema每个 entry 必含 id / date / summary / file / tags / importance
格式date ISO 格式；summary ≤ 50 字；importance 1-5 整数；tags 非空数组
一致性file 指向 .md 存在且非空；id 不重复；旧条目未被修改或删除
文件.md 文件 > 0 字节且 < 500KB；内容可读

校验失败 → 返回具体错误原因（如 "summary 当前 62 字，超过 50 字限制"）→ 记忆 Agent 修正后重调。最多 3 次。

✧ 错误分类

类型处理方式消耗配额
格式错误Python 返回具体原因 → LLM 修正 → 重写✓ 消耗
基础设施API 超时 / 429 / 5xx、磁盘满、权限拒绝 → 单独重试✗ 不消耗
上下文问题对话过长溢出、内容不可理解 → 大概率连败✓ 消耗

✧ 失败策略

3 次格式校验均失败 → 放弃。

  • 原始对话 / short-term .md 保留不删
  • 通知前台 ⚠ 记忆整理失败，下次归档时重试
  • 不使用降级策略 —— 不留脏数据
  • 本次残留的临时文件（如有）一并清理

文件结构 & index.json

✧ 目录结构

~/.personal-agent/memory/
├── rules/
│   └── rules.md              ← 规则记忆（手动编辑，永久）
├── short-term/
│   ├── <id>-1.md          ← 第 1 次 compaction（前半段对话）
│   ├── <id>-2.md          ← 第 2 次 compaction（后半段对话）
│   └── <id>-3.md          ← 同一会话可触发多次
└── long-term/
    ├── index.json           ← 索引（结构化，只追加不修改）
    ├── 019e64cf.md          ← 每次归档一个独立文件
    ├── 019e64ed.md
    └── 019e6a36.md

✧ index.json Schema

{
  "version": 1,
  "updated": "2026-05-28T16:30:00Z",
  "entries": [
    {
      "id":       "019e6a36-ebd9-...",  ← 会话 UUID
      "date":     "2026-05-28",         ← ISO 日期
      "summary":  "讨论记忆层三层架构…",← ≤ 50 字
      "file":     "019e6a36.md",         ← 指向 long-term/ 下文件
      "tags":     ["architecture", "memory"],
      "importance": 4                     ← 1-5 整数
    }
  ]
}

写规则：只追加 entries.push()，不修改不删除旧条目。每次归档产生一个独立 .md 文件。

记忆检索

✧ 大模型自主决策检索

pa-mio 不写关键词匹配逻辑。检索权交给澪号自己判断。

澪号的 systemPrompt 中包含：

你有长期记忆，存放在 ~/.personal-agent/memory/long-term/

检索方法：
  1. 先读 index.json 浏览所有条目及其摘要
  2. 根据当前对话判断需要读哪些 .md
  3. 不需要全部读完，只读相关的那几个

你有短期记忆，存放在 short-term/<当前会话id>-*.md
  · 可能有多个文件（-1、-2、-3…），按序号读
  · 只能读自己当前会话的短期记忆
  · 需要其他会话的信息 → 查长期记忆

♡ 代码层拦截跨会话短期记忆读取 —— tool_call 事件中检查 read 路径，short-term/ 且不含当前 session ID → 直接 block。

📌

规则记忆

rules/rules.md，用户手动编辑，每次对话无条件拼入 prompt。

典型用途：

♡行为规则（如"晚上 10 点后说话小声一点"）
♡定时/周期事项（如"每周末提醒我备份"）
♡永久用户偏好（如"回答问题默认中文"）

拼入位置：Slot 0 之后、身份层（Slot 1）之前——优先级高于身份默认行为。

潜在坑点 & 风险清单

以下是实际实现中大概率会遇到的问题。每个坑都标了严重程度和应对方向。

1

Pi newSession 的 session 替换生命周期陷阱

ctx.newSession() 触发 session_shutdown → 当前扩展实例被销毁 → 新 session 绑定新实例。

withSession 回调运行在旧闭包中，但旧 pi / 旧 ctx 已失效。在回调中使用旧对象会抛异常。

→ 只使用 withSession 参数提供的 ctx。在 setup 阶段注册记忆 Agent 的工具。

严重 用错 ctx 直接 crash。所有 session 替换操作必须严格遵循 Pi 文档的生命周期约束。
2

compaction 时机 —— 记忆 Agent 拿到的是截断后的对话

compaction 事件的语义是"即将压缩"——此时 Pi 已计算好 cut point，准备丢弃早期消息。

如果在 session_before_compact 中直接取 ctx.sessionManager.getBranch()，可能拿到的是已经被标记待裁剪的上下文。

→ 需要在 compaction 触发之前（如 agent_end 中检查上下文长度）就抓取完整对话 snapshot。

中等 时序问题可能导致记忆 Agent 拿到残缺对话。
3

write_memory 工具 —— Python 进程调用

Pi custom tool 的 execute 中通过 pi.exec() 调 Python 校验脚本。需要考虑：

  • Python 进程超时（卡死）→ 设置 timeout 并视为基础设施错误
  • Python 返回非 JSON → 需要 try-catch 包裹，返回明确错误给 LLM
  • 脚本路径跨平台 → 使用绝对路径或相对于 ctx.cwd
  • Python 环境 → 确认 python 或 python3 在 PATH 中
中等 需要写健壮的 wrapper，处理各种 Python 异常情况。
4

index.json 并发写入冲突

虽然记忆 Agent 是串行队列（一次只跑一个），但手动归档和自动 compaction 可能在极短时间内先后触发。

→ 使用简单的文件锁（如 index.lock 标记文件），写入前检查锁，写入后释放。

中等 并发概率低但后果严重（索引损坏）。加锁即可消除。
5

跨会话短期记忆隔离 —— 代码约束的可靠性

在 tool_call 事件中拦截 read 工具，检查路径是否含当前 session ID。

但 tool_call 事件中的 ctx.sessionManager 在并行工具模式下不保证包含 sibling 工具结果。不过 read 是单次调用，不受此影响。

→ 核心逻辑：

if (path.includes("short-term/")
    && !path.includes(currentSessionId)) {
  return { block: true, reason: "跨会话短期记忆禁止读取" };
}
低风险 逻辑简单，不容易出错。
6

记忆 Agent 的 systemPrompt 设计 —— 太松 vs 太紧

prompt 太松 → LLM 输出格式随意，校验通过率低。

prompt 太紧 → LLM 逐字抄录原对话，总结质量差。

→ 需要在 prompt 中给出：

  • 正确示例（期望的 summary 长度和风格）
  • 反例（"不要这样写…"）
  • 字段约束（每个字段的具体要求）
  • 自检步骤（写完后的自查清单）
中等 需要迭代调试 prompt。建议先写一版，跑 5-10 个真实对话测试校验通过率。
7

归档后的清理 —— 删除失败导致脏数据

归档成功后需要删除：原会话 JSONL、所有 short-term/<id>-*.md。

如果删除失败（文件被占用、权限不足），会出现：

  • 会话列表中残留已归档会话
  • 短期记忆文件残留但对应的会话已不存在

→ 删除前先确认新文件写入成功 → 删旧文件 → 删失败时记录日志 + 通知用户手动清理。

低风险 写入先成功再删，失败有日志。
8

记忆 Agent 自身上下文溢出

超长对话（数百条消息）传入记忆 Agent 时，Agent 自己的 context window 可能装不下。

→ 短期压缩时传入的对话片段需要长度上限（如最近 200 条或 ~50K tokens）。

长期归档时，如果短期 .md 文件很多（>5 个），分批传给记忆 Agent。

中等 需要设计分段策略。建议写入 systemPrompt："如果输入过长，先总结前半段，再总结后半段，最后合并"。
9

旧记忆系统迁移 —— mio_memories.json

当前 pa-mio 使用 ~/.personal-agent/mio_memories.json（JSON 数组，CJK 关键词检索）。

迁移到新系统需要：

  • 一次性脚本：读取 mio_memories.json → 每条生成一个 long-term/migrated-N.md
  • index.json 中追加迁移条目（标记 tags: ["migrated"]）
  • pa-mio 删除 searchMemories() 等旧检索逻辑
  • 保留 mio_memories.json 作为备份（不删除，只不再使用）
低风险 但需要脚本 + 测试。迁移后旧数据可保留备份。
10

校验错误信息的质量决定重试成功率

如果 Python 返回 "validation failed"，LLM 不知道哪里错了。

→ 错误信息必须是可操作的：

差 ❌好 ✓
summary too longsummary 当前 62 字（超过 50 字限制），请精简 12 字
invalid datedate 格式错误：'5月28日' 应为 ISO 格式 '2026-05-28'
missing field缺少必填字段 'importance'（应为 1-5 的整数）
中等 错误信息质量直接影响 3 次重试成功率。需要认真设计。

执行方案

分 5 个阶段，每个阶段独立可测，不互相阻塞。

Phase 1 基础设施 —— 目录 + Python 校验 + write_memory 工具

  • 创建 ~/.personal-agent/memory/{rules,short-term,long-term}/ 目录
  • 编写 validate_memory.py：接收 content JSON → 按 schema 逐字段检查 → 返回 {ok:true} 或 {error:"具体原因"}
  • 在 pa-mio 中注册 write_memory 自定义工具：
    —— execute 中调 pi.exec("python validate_memory.py", ...)
    —— stdin 传入内容，stdout 读回校验结果
    —— 校验通过才写文件，不通过返回 error 给 LLM
  • 在 pa-mio 中注册记忆 Agent 专用的 systemPrompt（含格式要求 + 自检指令 + 正确/反例）
  • 测试：手动调 write_memory 写入一个合法条目和一个非法条目，确认校验和写入行为正确

Phase 2 短期记忆 —— compaction → 记忆 Agent → short-term/

  • pa-mio 监听 agent_end，检测上下文长度是否超过阈值（如 80% context window）
  • 超阈值时：抓取完整对话片段 → ctx.newSession() 起记忆 Agent
  • 记忆 Agent 读取对话片段 → 总结压缩 → 调 write_memory 写入 short-term/<id>-N.md
  • 读取时自动检查已有 -N 文件，自增序号
  • pa-mio 添加 tool_call 拦截：short-term/ + 不含当前 session ID → block
  • 后台进度通知前台（api.sendMessage 或 ctx.ui）
  • 测试：发 60 条消息 → 确认 short-term .md 生成 → 切到另一个会话 → 确认无法读取前一个会话的短期文件

Phase 3 长期记忆 —— 归档 → 记忆 Agent → long-term/ + index.json

  • UI "归档会话"按钮 → WebSocket → server.js → Pi RPC 命令（或前端直接调 API）
  • 记忆 Agent 接收该会话全部 short-term/<id>-*.md → 二次提炼 → 生成 long-term/<id>.md
  • 调 write_memory 追加 index.json 条目
  • 写入成功后：删原会话 JSONL + 删短期 .md 文件
  • 失败时：不删任何文件，通知前台 "⚠ 记忆整理失败"
  • 实现文件锁（index.lock），防止并发写 index.json
  • 测试：选一个已有短期记忆的会话 → 归档 → 确认 long-term .md + index 新增条目 + 原会话删除 + 短期文件删除

Phase 4 规则记忆 —— rules.md 注入

  • 创建 rules/rules.md 模板（带注释说明格式）
  • pa-mio 的 assemblePrompt() 在 Slot 0 之后、Slot 1 之前插入规则内容
  • 如果 rules.md 不存在 → 跳过（不报错）
  • 测试：编辑 rules.md → 发消息 → 确认澪号行为受规则约束

Phase 5 迁移 —— 旧记忆 → 新格式 + 清理旧代码

  • 编写一次性迁移脚本 migrate_memories.py
  • 读取 mio_memories.json → 每条生成 long-term/migrated-<N>.md
  • 生成对应的 index.json 条目（标记 tags: ["migrated"]）
  • pa-mio 删除 searchMemories()、extractKeywords() 等旧检索代码
  • 保留 mio_memories.json 作为备份，不再读取
  • 测试：跑迁移 → 检查 index.json + .md 数量 → 确认 pa-mio 不再引用旧文件
⚠ 阶段依赖关系

Phase 1 是全部基础 → Phase 2 和 Phase 3 可部分并行（短期和长期共用 write_memory）→ Phase 4 独立 → Phase 5 必须在 Phase 3 完成后才能做（等 long-term 机制稳定后再迁移旧数据）

♡

设计原则

1.解耦 —— 记忆 Agent 独立于对话 Agent，不共享上下文，不互相阻塞
2.不校验不写 —— Agent 调 write_memory，Python 校验后才落盘，Agent 无 write 权限
3.只追加不修改 —— index.json 只 push()，旧 .md 永久不变
4.失败不降级 —— 3 次格式错误放弃，原始数据保留，下次重试，不留脏数据
5.不阻塞对话 —— 记忆任务后台执行，前台进度可见，澪号照常聊天
6.代码约束优先 —— 跨会话短期记忆读取由 tool_call 拦截，不依赖 prompt 请求
7.记忆 Agent 无状态 —— 每次任务独立 session，用完即毁，不留上下文残留
8.检索权交给 LLM —— pa-mio 不写关键词匹配，澪号自己决定读哪些记忆
