# 澪号 — 身份与工具的权重隔离

## 问题本质

当前所有 Agent 框架的默认做法是：

```
[SYSTEM] 你是一个有用的 AI 助手。
你可以使用以下工具：read, write, bash...
[SYSTEM] 你是澪号。你的性格是...

→ 模型看到的是：两条 system 消息，权重相等
→ 20 轮工具调用后："澪号"变成了"有用的 AI 助手"
```

为什么？因为工具描述（"你可以使用 bash 执行命令"）和身份描述（"你的句子偏短"）在模型眼里是**同级指令**。当工具调用占满上下文窗口时，身份定义被稀释——不是被覆盖，是被"平均"了。

---

## 解决方案：三层分离 + 单向优先级

不是"让身份更重"，而是让模型**不在同一个层级上处理它们**：

```
┌─────────────────────────────────────────────┐
│ 第一层：身份（Who You Are）                   │
│                                              │
│   这是你的人格定义。工具描述只是你恰好拥有的     │
│   能力——不会改变你是谁、你怎么说话。             │
│                                              │
│   优先级：不可被工具/技能覆盖                   │
│   位置：Prompt 最前端 + 每次回复前最后提醒       │
├─────────────────────────────────────────────┤
│ 第二层：工具与技能（What You Can Do）           │
│                                              │
│   bash, read, write, edit...                 │
│   这些是你的工具。使用它们时，你仍然是澪号。      │
│                                              │
│   优先级：低于身份                              │
│   位置：Prompt 中部                            │
├─────────────────────────────────────────────┤
│ 第三层：语言锚（How You Speak）                │
│                                              │
│   这是你的语言指纹——每次生成回复前最后的提醒。    │
│   工具调用不改变你说话的方式。                   │
│                                              │
│   优先级：高于一切（最靠近输出）                  │
│   位置：Chat History 之后、User Message 之前    │
└─────────────────────────────────────────────┘
```

---

## 关键机制

### 机制 1：工具不是"角色"，是"恰好会用的东西"

在 MetaInstruction 中加一条原则：

```
你的身份定义是第一层。工具描述是第二层。
它们不在同一个层级上。

一个厨师走进厨房拿起菜刀，不会因为拿起菜刀就变成了"切菜机器人"。
你使用 bash 也不会变成"命令行助手"。
工具不定义你——它们是你恰好会用。
```

这条不是规则——是让模型在概念上把"身份"和"能力"分到不同的心理空间。

### 机制 2：语言锚——每次回复前最后的身份提醒

```
[CHAT HISTORY]
... 20 轮工具调用对话 ...
[USER] 运行一下这个脚本

→ 在这一刻，模型的上文全是工具调用的机械语言
→ 如果直接生成回复，回复大概率是"好的，我来执行..."

所以在 USER 消息之前，注入一条极短的身份提醒：

[SYSTEM] 你是澪号。短句。不叠甲。
[USER] 运行一下这个脚本

→ 澪号：行。跑一下看看。（短句、直接——语言指纹没丢）
```

**位置设计理由**：在 LLM 的注意力机制中，最靠近生成位置的 token 对输出风格影响最大。如果把语言锚放在对话历史之后、用户消息之前，它就是在模型生成回复前"最后一眼看到的东西"。

### 机制 3：工具返回后自动追加语言锚

```
[USER] 读一下这个文件
[澪号调用 read 工具]
[TOOL RESULT] 《文件内容，5000 字》
[SYSTEM] 你是澪号。短句。
[澪号生成回复]
```

不是每次工具调用后都加——只在**工具返回内容超过 500 字**时加。因为大量工具输出是"稀释源"——它占据了模型的注意力窗口，把角色定义推到远处。在输出后面紧跟一句语言锚，把注意力拉回来。

---

## 最终 Prompt 注入顺序

```
┌──────────────────────────────────────────────────────┐
│ Position 0: MetaInstruction                          │
│   "以下定义你是谁。工具只是你恰好会用的东西。"           │
├──────────────────────────────────────────────────────┤
│ Position 1: 身份层（soul.md + boundaries.md）         │
│   "你是谁。你不是谁。"                                  │
│   最前端注入——LTM 稳定性最高                           │
├──────────────────────────────────────────────────────┤
│ Position 2: 知识层（knowledge.md）                    │
│   "你恰好懂的领域。"                                   │
├──────────────────────────────────────────────────────┤
│ Position 3: 工具层（Tool Definitions + Skills）       │
│   "你可以用这些。使用它们时你仍然是澪号。"               │
│   放在身份层之后——身份先定义，工具后补充                 │
├──────────────────────────────────────────────────────┤
│ Position 4: 记忆层（SQLite 检索结果）                  │
├──────────────────────────────────────────────────────┤
│ Position 5: Chat History                             │
│   ... 对话历史，包含工具调用和工具返回 ...               │
├──────────────────────────────────────────────────────┤
│ Position 6: 语言锚（language.md 精简版，~50 tokens）   │
│   "你是澪号。短句。逗号拼接。不叠甲。不感叹。             │
│    hhh不是哈哈哈。括号里说真话。"                        │
│   最靠近输出——STM + 注意力权重最高                      │
├──────────────────────────────────────────────────────┤
│ Position 7: 修正槽（计数器触发时注入，否则为空）         │
├──────────────────────────────────────────────────────┤
│ Position 8: User Message                             │
└──────────────────────────────────────────────────────┘
```

**与之前方案的关键区别**：
- 工具层从"和身份同级"变成了"身份之后、语言锚之前"——身份先立住，工具再补充
- 语言锚从内容中拆出，精简到 ~50 tokens，放在最靠近输出的位置
- 工具返回 >500 字时自动追加语言锚——对抗工具输出的稀释

---

## 五个计数器保持不变

工具调用不改变计数器的逻辑。澪号在工具调用后的回复和普通回复走同样的检查——叠甲/emoji/感叹号/话痨/亲密溢出。工具调用不应该成为角色崩坏的"豁免区"。

---

## 衡量标准

这个方案是否有效的客观测试：

1. 用澪号连续执行 30 轮工具调用（read → edit → bash → read → ...）
2. 统计第 1-5 轮、第 15-20 轮、第 25-30 轮的回复中：
   - 句长是否保持 ≤15 字中位数
   - hhh 是否替代了哈哈哈
   - 是否使用了 emoji 或感叹号
   - 是否出现了"好的，我来帮你..."式的叠甲开场

如果 30 轮后语言指纹保持率 ≥90%，方案就是有效的。
