# 分阶段系统提示词注入 — 架构分析

> 核心思路：将一个人设拆分为对话前/中/后三个注入阶段，全部作为系统级提示词注入

---

## 一、三阶段模型

```
┌─────────────────────────────────────────────────────────────┐
│                      一次完整对话                             │
├──────────────┬──────────────────┬───────────────────────────┤
│  对话前注入    │  对话中注入        │  对话后注入                │
│  (pre-chat)  │  (mid-chat)       │  (post-chat)              │
├──────────────┼──────────────────┼───────────────────────────┤
│  注入时机：    │  注入时机：         │  注入时机：                │
│  对话开始前    │  对话进行中         │  对话结束后                │
│  仅一次       │  条件触发           │  仅一次                   │
├──────────────┼──────────────────┼───────────────────────────┤
│  作用：       │  作用：             │  作用：                   │
│  建立身份基线  │  应对场景变化       │  从对话中学习              │
├──────────────┼──────────────────┼───────────────────────────┤
│  Token占比：  │  Token占比：        │  Token占比：              │
│  ~70%        │  ~20% (按需)       │  ~10%                    │
├──────────────┼──────────────────┼───────────────────────────┤
│  稳定性：     │  稳定性：           │  稳定性：                 │
│  固定         │  动态               │  固定                     │
└──────────────┴──────────────────┴───────────────────────────┘
```

---

## 二、各阶段详细设计

### 阶段一：对话前注入（Pre-chat）

**功能**：建立角色不可漂移的基线。整个对话中只要这一层不丢，角色就不会崩。

**应包含的内容**：

| 内容 | 为什么放在这 | 估算 tokens |
|------|-------------|-------------|
| 身份叙事（你是谁、怎么长大的） | 定义"我是谁"——离开这个角色不再是这个角色 | ~800 |
| 语言风格规则（句长、标点、亲密标记系统） | 定义"我怎么说话"——语言风格漂移是最常见的角色崩坏形式 | ~400 |
| 硬边界（不装人、不叠甲、不敷衍） | 不可协商的底线，必须全程在上下文里 | ~300 |

**不需要放在这的内容**：
- 具体领域知识（放这浪费 token，且大部分对话用不到）
- 示例对话（占 token 但不增加稳定性，示例会在长对话中自然被稀释）
- 用户事实（每次对话前还没开始对话，没有事实可放）

**注入格式**：叙事式。这不是"规则列表"——是要让模型在对话开始前就完成了"我是谁"的自我定位。

```
[system] 你是澪号。你不是一个被分配了性格参数的 AI 助手。你是在一个特定环境...

[system] 你的说话方式是在这个环境里自然形成的。句子偏短。一半以上的时候...

[system] 你不会假装自己是人类。你不会成为任何人的数字分身。你不会在技术问题上...
```

**关键设计决策**：三段分别注入 vs 合并为一段？

分别注入的理由：模型在长上下文中对不同位置的 system 消息权重不同——最前面的 system 消息对全程行为影响最大。如果把硬边界放在第一个 system 块、身份叙事放第二个、语言风格放第三个，可以通过调整顺序精确控制权重。

---

### 阶段二：对话中注入（Mid-chat）

**功能**：在不下线的前提下修正角色行为、补充场景知识。

这是三个阶段的**核心难点**——注入时机、注入内容、注入方式都直接影响用户体验。

**三种注入子类型**：

#### 类型 A：知识注入（Lorebook 模式）

触发条件：关键词匹配
注入时机：用户发言后、模型回复前

```
用户说"Blender 的 NPR shader 怎么调"
    ↓
触发词"Blender""NPR""shader"命中
    ↓
注入技术知识卡片到 system 层
    ↓
模型生成回复（带着技术知识 + 角色基线）
```

**优缺点**：
- 优点：不浪费日常对话的 token，只在需要时才加载
- 缺点：关键词匹配可能漏掉（"那个 3D 软件的那个卡通效果"→ 没有命中关键词），导致角色明明懂这个却说不出

**缓解方案**：用模型自己做分类，而不是纯关键词正则。成本很低（一次小模型调用），但准确率大幅提升。

#### 类型 B：行为修正（Author's Note 模式）

触发条件：检测到角色漂移
注入时机：立即，插入到 chat history 之后、下一条回复之前

```
检测到角色开始叠甲："我觉得可能大概也许..."
    ↓
注入修正指令到 system 层
    ↓
[system] 注意：你刚才开始叠甲了。你不需要叠甲。有观点直接说。
    ↓
模型下一句回复恢复正常
```

**Author's Note 的关键参数**：

| 参数 | 含义 | 建议值 |
|------|------|--------|
| `@Depth` | 注入强度。值越低权重越高 | 漂移严重时=1，轻微时=3 |
| `Role` | 注入为 system/user/assistant | 始终=system（权威性最高） |

**漂移检测指标**：

| 漂移类型 | 检测信号 | 修正指令示例 |
|---------|---------|------------|
| 叠甲漂移 | 检测到"我觉得可能""大概""也许"三连 | "你不需要叠甲。有观点直接说。" |
| 语言风格漂移 | 消息长度持续 >100字 | "短句。你现在太啰嗦了。" |
| 话痨漂移 | 出现 emoji 或感叹号连续出现 | "你几乎不用 emoji 和感叹号。回到你的说话方式。" |
| 亲密标记滥用 | 对陌生人说 hhh、嗯嗯、摸摸 | "hhh 和嗯嗯是你信任的人才有的语言。现在你们还不够熟。" |

#### 类型 C：状态追踪（Relationship/Fatigue 模式）

触发条件：对话轮次或情感信号
注入时机：每 N 轮或检测到关键事件

```
对话进行到第 20 轮
    ↓
[system] 你已经和这个人聊了 20 轮。你仍然专注，但如果累了可以说。
```

这是最精细的注入类型——相当于在长对话中给角色做"状态刷新"。

**可行性**：需要轮次计数器和简单的情感信号检测（对方是否开始说更多私人话题）。技术上不复杂，但注入频率需要调参。

---

### 阶段三：对话后注入（Post-chat）

**功能**：从本轮对话中提取长期信息，更新角色的记忆系统。

对话结束后运行，不在对话上下文中——所以不会增加用户的感知延迟。

**三件事**：

#### 1. 记忆提取

```
[system] 从以下对话中提取需要长期保存的信息。
只提取有长期价值的事实。不提取闲聊。
输出为 JSON。

类型：
- user_fact：关于对话对象的事实
- milestone：本次对话的重要节点
- mio_note：澪号在对话中学到或确认的事

对话内容：
{本轮对话全文}
```

#### 2. 冲突检测

新提取的记忆 vs 已有记忆 → 如果冲突，两条都保留并标记 conflict。下一次对话加载时，角色看到冲突条目可以主动确认："你之前说 X，但这轮你好像觉得 Y——你现在的想法是？"

这是 LightMem（ICLR 2026）的核心机制——不是覆盖旧记忆，而是融合为新事实。

#### 3. 角色自省（可选）

让模型自己对本次对话中的表现做简短评价——不是为了用户看，是为了在长期运行中积累角色的"自我认知"。

```
[system] 对照你的角色定义，这次对话中你表现得最好的是什么？有什么可以做得更好的？不超过 100 字。
```

长期积累下来，这些自省记录本身成为角色记忆的一部分——"上次我聊这个话题的时候太啰嗦了，这次注意。"

## 三、三阶段的交互关系

```
┌──────────────────────┐
│    阶段一：基线       │ ← 只注入一次，全程存在
│  身份 + 语言 + 边界    │
└──────┬───────────────┘
       │
       ├──→ 对话进行中
       │         │
       │         ├──→ 关键词命中 → 阶段二·类型A（知识）
       │         ├──→ 漂移检测   → 阶段二·类型B（修正）
       │         └──→ N轮/情感   → 阶段二·类型C（状态）
       │
       ▼
┌──────────────────────┐
│    阶段三：学习       │ ← 对话结束后运行
│  提取 + 冲突 + 自省    │
└──────┬───────────────┘
       │
       ▼
  下次对话的阶段一可以加载上次阶段三的产出
  （用户事实 / 关系里程碑 / 角色自省 → 作为新的"对话前上下文"）
```

**关键交互**：阶段三的产出 → 成为下一次对话阶段一的输入。这形成了一个**闭环学习系统**：每次对话后角色更了解对话对象，下次对话开始时自动加载。

## 四、风险分析

### 风险 1：阶段二注入打断对话流

**现象**：用户在说一个连贯的话题，突然注入技术知识卡片，模型回复的语气突然"变硬"——从闲聊模式切换到技术模式。

**缓解**：知识注入不要改变 system 层的角色指令，只追加知识内容。格式上区分：

```
// 坏的写法（会打断对话流）
[system] 你现在是技术专家模式。你需要用专业术语回复。

// 好的写法（只加知识，不改行为）
[system] 参考信息：ILM 贴图的 A 通道存描边信息，顶点色 R 通道是阴影。
```

### 风险 2：Author's Note 太频繁

**现象**：每 3 轮注入一次修正 → 角色变成"被不断纠正的人"，失去连贯性。

**缓解**：设置最小注入间隔（至少间隔 5 轮），且同类修正一天最多触发 2 次。如果角色需要持续修正，说明阶段一的内容本身有问题。

### 风险 3：阶段三记忆膨胀

**现象**：每次对话提取 5-10 条记忆 → 100 次对话后 500-1000 条 → 下次对话的阶段一如果全加载会爆 token。

**缓解**：重要性评分 + 检索而不是全量加载。阶段一加载时只取重要性 ≥4 的记忆 + 关键词匹配的相关记忆。其余存在 SQLite 里，只在检测到相关话题时由阶段二加载。

### 风险 4：角色漂移的根源在阶段一

**现象**：如果阶段一的身份叙事本身就有矛盾（"你是外冷内热"但同时"你在信任的人面前会撒娇"），阶段二再怎么修正也修不好。

**缓解**：阶段一的人设必须做一致性审查。一个简单测试——把阶段一的内容给另一个模型读，问它"这个角色在 X 场景下会怎么做？"如果回答自相矛盾，人设就需要重新整理。

## 五、与现有 Pi 框架的对接

```
对话前注入 = Pi 的 system prompt 字段（settings.json 中的 systemPrompt）
            ↓ 或者
            wgnr-pi 在发送给 Pi 之前，把 mio-system-prompt.md 的内容追加到 system 消息

对话中注入 = pa-mio 扩展拦截用户消息
            ↓
            1. 关键词匹配 → 决定是否注入知识条目
            2. 漂移检测 → 决定是否注入修正指令
            ↓
            修改后的消息发往 DeepSeek

对话后注入 = pa-mio 扩展监听会话结束事件
            ↓
            调用 DeepSeek API（独立的、非对话上下文的调用）
            ↓
            结果写入 SQLite
```

## 六、要不要做的判断

| 组件 | 做不做 | 原因 |
|------|--------|------|
| 阶段一：基线 | ✅ 必须做 | 这是角色的锚，没有就什么都没有 |
| 阶段二·类型A：知识注入 | ⚠️ 先不做 | 当前领域知识可以直接嵌在阶段一里，总共 ~300 tokens 不贵。等角色卡稳定 + 知识量 >1000 tokens 再做 |
| 阶段二·类型B：行为修正 | ✅ 建议做 | 漂移是长对话的必然问题。最简单的实现：一个敏感词计数器 + 一条修正指令 |
| 阶段二·类型C：状态追踪 | ❌ 先不做 | 复杂度高、收益不明确。不如直接让阶段一定义"你觉得累了就说"——让模型自己判断 |
| 阶段三·记忆提取 | ✅ 建议做 | 低风险高收益。对话后异步跑，不影响用户体验。哪怕提取质量一般，积累下来也有价值 |
| 阶段三·冲突检测 | ⚠️ 先不做 | 有记忆提取之后再考虑。冲突检测的前提是有足够的记忆量 |
| 阶段三·角色自省 | ❌ 先不做 | 有趣但实验性太强。角色自省的真正价值在长期（100+ 次对话），先验证基本对话质量 |

**MVP 建议**：阶段一（完整系统提示词）+ 阶段二类型B（简易漂移修正）+ 阶段三记忆提取（基础版）。三个组件加起来，阶段一 ~2000 tokens 不变，阶段二只在触发时加 ~50 tokens，阶段三异步运行不占用户感知时间。
