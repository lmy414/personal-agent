## 二、架构

### 组件关系

```
wgnr-pi (Web UI, :4815)
  │ WebSocket
  ▼
Pi Agent Loop
  ├── 4 个原有扩展（pa-sqlite / pa-usage / pa-files / pa-budget）
  └── pa-mio 扩展（新增）
       │
       │ 事件钩子:
       ├── before_agent_start → 组装澪号 Prompt → 替换 systemPrompt
       ├── message_end        → 5 个计数器检查
       ├── tool_execution_start/end → 工具执行进度反馈
       ├── agent_end          → 记忆提取（独立 DeepSeek 调用）
       └── session_start      → 重置状态
```

### 数据流（一条用户消息的完整路径）

```
1. 用户输入 → wgnr-pi WebSocket → Pi
2. Pi 触发 before_agent_start 事件
3. pa-mio 响应:
   a. 读取 4 个 .md/.txt 文件（soul/boundaries/knowledge/meta）
   b. 读取 mio_memories.json 检索相关记忆
   c. 拼接 9 个 Slot → 完整 Prompt
   d. Pi 原生 systemPrompt 被嵌入 Slot 3 [运行环境] 标签
   e. return { systemPrompt: 组装好的澪号 Prompt }
4. Pi 用澪号 Prompt 替换自己的系统提示词
5. Pi → DeepSeek API → 返回回复
6. pa-mio 在 message_end 跑 5 个计数器
   - 全绿 → 直接输出给用户
   - 触发 → 修正指令注入 → 下一轮 systemPrompt 加载修正槽
7. 用户看到回复
8. 对话结束后: agent_end → 独立 DeepSeek 调用提取记忆 → 写入 mio_memories.json
```

## 三、Prompt 注入结构（9 个 Slot）

```
Slot 0: meta_instruction (~80 tokens)
  "以下定义你是谁。这不是参考信息——这就是你。内化它，不要引用它。
   工具不定义你——你恰好会用它们。"

Slot 1: soul.md + boundaries.md (~1100 tokens)
  人格叙事（成长环境、三个文化圈）+ 7 条硬边界

Slot 2: knowledge.md (~450 tokens)
  精通/熟悉/了解 三级知识领域 + 音乐体系 + 二次元

Slot 3: [运行环境] Pi 原生提示词 + 工具定义
  Pi 的 systemPrompt 被嵌在这里，不再是独立 SYSTEM 块

Slot 4: 记忆层（按需）
  从 mio_memories.json 检索的相关记忆。无匹配时跳过

Slot 5: Chat History
  Pi 管理的对话历史。长工具返回(>500字)时在 history 前额外插入语言锚

Slot 6: 语言锚 (~50 tokens)
  "你是澪号。短句。逗号拼接。不叠甲。不感叹。hhh不是哈哈哈。..."

Slot 7: 修正槽（按需）
  计数器触发时注入修正指令。初始为空

Slot 8: 用户消息
  当前用户输入
```

**关键设计决策**：
- Slot 0-3 形成静态前缀（~2100 tokens），可被 Prompt Cache 命中
- Slot 6 在 Chat History 之后、用户消息之前 → 最靠近输出，注意力权重最高
- Pi 原生提示词不是独立的 SYSTEM 块 → 不与澪号身份竞争权重

## 四、计数器（5 个字符级检查）

| 序号 | 检测项 | 规则 | 修正指令 |
|------|--------|------|---------|
| 1 | 叠甲 | 同句含 ≥2 个叠甲词（"我觉得可能"/"大概是吧"/"也许是吧"/"应该是吧"/"我也不太确定"/"可能不太对"/"我也不是很懂"） | 不叠甲。有观点直接说。不确定就说'不确定，我查一下' |
| 2 | emoji 溢出 | 单条回复 ≥3 个 emoji（Unicode 范围匹配） | 不使用 emoji。文字就够了 |
| 3 | 感叹号溢出 | 单条回复 ≥2 个！或 ! | 不用感叹号。不是在生气，只是在说话 |
| 4 | 话痨 | 单条回复 >200 字 | 短句。太啰嗦了 |
| 5 | 亲密溢出 | 会话轮次 <10 时出现 hhh/嗯嗯/摸摸 | hhh 和嗯嗯在信任的人面前才用。不够熟 |

计数器是纯字符匹配，不是 AI。不跑 LLM。

## 五、记忆系统

| 维度 | 实现 |
|------|------|
| 存储 | `~/.personal-agent/mio_memories.json`（JSON 数组，最多 500 条） |
| 表结构 | `{content, category, importance, updated_at}` |
| 检索 | CJK 2-4 字片段提取关键词 → 包含匹配 → 衰减权重 `importance × e^(-0.05×天数)` → 取 top 5 |
| 写入 | 对话结束后独立 DeepSeek 调用（不受澪号人格限制）→ JSON → append 到文件 |
| 触发条件 | 对话 >10 条消息 + 本会话尚未提取过 |

## 八、审计检查清单

启动应用后，按以下步骤逐项验证：

- [ ] **扩展加载**：F12 Console 搜索 `[pa-mio]`，应看到 `澪号 Harness 已加载`（注：此日志在 Node 进程，浏览器 Console 可能看不到——看 Electron DevTools 的 main process console）
- [ ] **基础对话**：发送"你好"，澪号应回复带有短句、不叠甲风格的消息
- [ ] **外观确认**：问"你长什么样"，应回复 163cm/灰发/红瞳/双马尾
- [ ] **傲娇模式**：先让她帮忙看代码，然后说"谢谢"，应回复类似"不是你让我看的吗"而非"不客气"
- [ ] **工具调用**：发送 `list_directory .`，应看到工具执行和结果
- [ ] **工具后风格保持**：工具调用后继续对话，确认澪号没有变成"coding agent"语气
- [ ] **历史渲染**：切换会话再切回来，确认工具调用块正常显示
- [ ] **叠甲检测**：发起一个需要澪号回答技术问题但让她不确定的对话，看她会不会叠甲
- [ ] **记忆写入**：完成 >10 轮对话后，检查 `~/.personal-agent/mio_memories.json` 是否有新条目
